                            1.  INTRODUCTION
================================================================================

1.1  Problem Definition
────────────────────────

Traditional database management demands specialised SQL knowledge, which
creates a significant barrier for business analysts, educators, students, and
non-technical users who need to interact with structured data.  Writing
correct SQL queries — from table creation to complex aggregations, joins, and
analytical visualisations — is error-prone, time-consuming, and requires
continuous reference to documentation.

Existing tools either:
  (a) provide a raw SQL console with no guidance, or
  (b) use rigid GUI form-builders that sacrifice flexibility.

Neither approach allows an end-user to describe what they need in plain
English and have the system autonomously write, validate, execute, and
visualise the results.

Invixa AI targets this gap by providing an intelligent, conversational
interface that translates natural-language prompts into actionable SQL
operations and dynamic data visualisations — all without the user ever
needing to write a single line of SQL.


1.2  Motivation
────────────────

The rapid advancement of Large Language Models (LLMs) such as DeepSeek has
opened new possibilities for human–computer interaction with structured data.
The key motivations behind this project are:

  •  Democratising Data Access
     Non-technical users (managers, analysts, students) should be able to
     query, mutate, and visualise database data without learning SQL.

  •  Reducing Development Overhead
     Small teams and solo developers can prototype database schemas and
     explore data faster by conversing with an AI assistant rather than
     manually writing boilerplate SQL.

  •  Real-Time Feedback Loop
     Traditional workflows involve writing a query, running it, copying
     results to a charting tool, and then iterating.  Invixa AI collapses
     this pipeline into a single conversational step: the user asks a
     question, the AI generates the query, executes it, and renders a
     chart — all within seconds.

  •  Privacy-First, Local-First Design
     Unlike cloud-based BI tools, Invixa AI runs entirely on the user's
     local machine.  The database (SQLite), the AI runtime (Ollama), and
     the web servers (Vite + FastAPI) all run locally — meaning sensitive
     data never leaves the device.

  •  Educational Value
     By showing the generated SQL alongside the results and explanation,
     the system serves as a learning tool for users who want to understand
     SQL progressively.


1.3  Objective of System
─────────────────────────

The primary objectives of Invixa AI are:

  1. To develop a full-stack web application that converts natural-language
     prompts into valid SQLite queries using a locally-hosted LLM.

  2. To provide real-time database management capabilities including table
     creation, data insertion, updating, deletion, and querying — all
     through a conversational chat interface.

  3. To generate dynamic, interactive data visualisations (bar, line, area,
     pie, scatter, and composed charts) from natural-language descriptions,
     rendered in real-time using Recharts.

  4. To implement a secure user authentication system with email/password
     registration, login, profile management, and password change features
     using bcrypt hashing.

  5. To maintain a persistent history of AI-generated charts and database
     table metadata so users can revisit, review, and re-render previous
     analytical work.

  6. To deliver a premium, modern, dark-themed user interface with
     glassmorphism aesthetics, micro-animations, and responsive layouts
     that work across screen sizes.

  7. To ensure complete data privacy by keeping all processing local — no
     data is sent to external cloud services.

2.1  Functional Requirements
─────────────────────────────

  FR-01  User Registration
         The system shall allow new users to register with a username, email
         address, and password.  Duplicate emails shall be rejected with
         a 409 Conflict error.

  FR-02  User Authentication
         The system shall provide email + password login.  On successful
         login, the user profile (without password) is returned to the
         client and stored in localStorage for session persistence.

  FR-03  Profile Management
         Authenticated users shall be able to update their username and role,
         change their password (requiring the current password for
         verification), and sign out.

  FR-04  Natural-Language to SQL Conversion
         The system shall accept a plain-English prompt (e.g., "Create a
         users table with name, email, and age") and generate valid SQLite
         SQL using the DeepSeek LLM via Ollama.

  FR-05  SQL Execution
         Generated SQL shall be executed against the SQLite database
         (sql_ai.db) in real-time.  The system shall return the query
         result, affected row count, and a plain-English explanation.

  FR-06  Database Preview
         The system shall provide live database preview showing all tables,
         their column structures, row counts, and data rows.  The preview
         auto-refreshes after each mutation.

  FR-07  Natural-Language Chart Generation
         The system shall accept a plain-English chart request (e.g., "Show
         me a pie chart of users by role"), generate the appropriate SQL
         query and chart configuration, execute the query, and return
         structured chart data.

  FR-08  Chart Rendering
         The frontend shall dynamically render six chart types — bar, line,
         area, pie, scatter, and composed — using the Recharts library
         based on the AI-generated specification.

  FR-09  Chart History Persistence
         Every generated chart shall be saved to the application database
         (app.db) with its title, chart type, colour, SQL query, axis keys,
         and creation timestamp.

  FR-10  Chart History Retrieval
         The system shall retrieve and display the most recent saved charts
         on the Account/Settings page, re-running the original SQL query
         against the live database for up-to-date data.

  FR-11  Dashboard Home Page
         The system shall present a dashboard home page showing database
         statistics (total tables, total rows, active engine), a quick
         AI prompt bar, and a list of recently modified tables.

  FR-12  Greeting Detection
         When the user sends a conversational message (greeting, general
         question), the system shall detect that no SQL is needed and
         respond conversationally without executing any query.

  FR-13  Error Handling
         The system shall gracefully handle: empty databases, invalid LLM
         responses, malformed JSON, SQL execution failures, missing schema
         fields, and network errors — displaying user-friendly messages.

3.2  Modules
─────────────

The system is organised into the following modules:

  MODULE 1:  Authentication Module
  ─────────────────────────────────
    •  User Registration (POST /app/auth/register)
    •  User Login (POST /app/auth/login)
    •  Session Management (localStorage-based, stateless)
    •  Password Hashing (bcrypt with 72-byte truncation)
    •  User Logout (POST /app/auth/logout)

  MODULE 2:  SQL AI Assistant Module
  ────────────────────────────────────
    •  Schema Reflection — dynamically reads the current database schema.
    •  Prompt Engineering — constructs a system prompt with schema context.
    •  LLM Integration — sends the prompt to DeepSeek via Ollama.
    •  SQL Validation — detects whether the response is SQL or plain text.
    •  SQL Execution — executes the generated query via SQLAlchemy.
    •  Explanation Generation — asks the LLM for a plain-English explanation.
    •  Table Metadata Sync — updates app.db with table stats after mutations.

  MODULE 3:  Analytics / Chart Generation Module
  ────────────────────────────────────────────────
    •  Schema + Sample Reflection — reads schema and up to 5 sample rows.
    •  Chart Prompt Engineering — constructs a constrained system prompt.
    •  JSON Parsing & Validation — strips markdown fences, validates fields.
    •  SQL Execution — runs the chart's data query.
    •  Numeric Coercion — converts string numbers to floats for Recharts.
    •  Chart Persistence — saves chart metadata to app.db (SavedChart).
    •  Chart History API — retrieves and re-runs saved charts for live data.

  MODULE 4:  Database Preview Module
  ────────────────────────────────────
    •  Table Listing (GET /db/tables) — lists all tables with row/col counts.
    •  Table Data Retrieval (GET /db/table/{name}) — full column + row data.
    •  Auto-Refresh — the preview re-fetches after each SQL mutation.

  MODULE 5:  User Profile & Settings Module
  ───────────────────────────────────────────
    •  Profile Display — avatar, username, email, role, join date.
    •  Profile Editing — inline edit for username.
    •  Password Change — requires current password verification.
    •  Database Statistics — total tables, rows, engine info.
    •  Chart History Display — mini-charts with expand-to-modal feature.

  MODULE 6:  Dashboard / Home Module
  ────────────────────────────────────
    •  Greeting with user's name.
    •  Quick Prompt Input — redirects to Analytics tab.
    •  Database Overview Cards — total tables, rows, active engine.
    •  Recently Modified Tables — table list with column/row counts.

  MODULE 7:  Frontend UI/UX Module
  ──────────────────────────────────
    •  Dark-themed design with glassmorphism.
    •  Responsive layout (desktop-optimised, mobile-friendly).
    •  Micro-animations using Framer Motion and CSS.
    •  Typewriter text effect for AI responses.
    •  Markdown rendering with syntax-highlighted tables and code.
    •  Sticky header with tab-based navigation.

  Use Case Descriptions:

    UC1 – Register Account:  User provides username, email, and password.
           System hashes the password with bcrypt and stores the record
           in app.db.

    UC2 – Login / Logout:    User submits email and password.  System
           verifies credentials, returns user profile.  Logout is
           stateless (client clears localStorage).

    UC3 – Manage Profile:    User views and edits username.  User can
           change password (requires current password verification).

    UC4 – Send NL Prompt:    User types a natural-language request.
           System constructs a context-aware prompt with the current
           database schema and sends it to the DeepSeek LLM via Ollama.

    UC5 – Execute SQL:       The LLM-generated SQL is run against
           sql_ai.db.  Results (rows or row count) are returned.
           An explanation is generated by a second LLM call.

    UC6 – Preview Database:  User views the list of tables, selects one,
           and sees all columns and rows in a live preview panel.

    UC7 – Request Chart:     User describes a chart in plain English.
           System generates chart spec (JSON) via LLM, executes the
           SQL query, and returns structured data for rendering.

    UC8 – View Chart History: User visits the Account page and sees
           previously generated charts, rendered with live data.

    UC9 – View Dashboard:    User lands on the Home page and sees
           database stats, prompt input, and recently modified tables.

3.3.5  Algorithm — Natural-Language to SQL Pipeline
────────────────────────────────────────────────────

  ALGORITHM:  GenerateAndExecuteSQL

  Input:   user_prompt (string), user_id (optional integer)
  Output:  JSON response with SQL, result data, and explanation

  BEGIN
    1.  REFLECT the current database schema from sql_ai.db
        schema ← MetaData.reflect(engine)
        schema_info ← format each table's columns and types as string

    2.  CONSTRUCT system instruction
        system_prompt ← concatenate(role_instructions, schema_info, rules)

    3.  CALL the LLM
        response ← ollama.chat(model="deepseek-v3.1:671b-cloud",
                                system=system_prompt,
                                user=user_prompt)
        generated_text ← response.message.content.strip()

    4.  CHECK if response is SQL
        IF is_sql(generated_text) = FALSE THEN
            RETURN { status: "chat", explanation: generated_text }
        END IF

    5.  GENERATE explanation
        explanation ← ollama.chat("Explain: " + generated_text)

    6.  EXECUTE the SQL
        WITH engine.connect() as connection:
            result ← connection.execute(text(generated_text))
            connection.commit()

            IF result.returns_rows THEN
                content ← list of row dictionaries
                message ← "Data fetched successfully"
            ELSE
                content ← { rows_affected: result.rowcount }
                message ← "Database updated successfully"
            END IF

    7.  SYNC table metadata
        IF NOT result.returns_rows THEN
            sync_tables_to_app_db(user_id)
        END IF

    8.  RETURN {
            status: "success",
            executed_sql: generated_text,
            message: message,
            explanation: explanation,
            content: content
        }
  END


  ALGORITHM:  GenerateChartFromPrompt

  Input:   user_prompt (string), user_id (optional integer)
  Output:  JSON response with chart specification and query data

  BEGIN
    1.  REFLECT schema and collect sample rows (up to 5 per table)
        schema ← get_schema_and_sample()
        IF schema is empty THEN
            RAISE HTTPException(422, "Database is empty")
        END IF

    2.  CONSTRUCT system prompt with schema, chart constraints,
        and required JSON shape

    3.  CALL the LLM
        raw ← ollama.chat(system=system_prompt, user=user_prompt)

    4.  STRIP markdown fences
        clean ← strip_markdown_fences(raw)

    5.  PARSE JSON
        spec ← json.loads(clean)
        IF parse fails THEN RAISE HTTPException(422)

    6.  VALIDATE required fields
        required ← [chart_type, title, sql, x_key, y_key]
        IF any missing THEN RAISE HTTPException(422)

    7.  CONSTRAIN chart_type to allowed set
        IF spec.chart_type NOT IN {bar,line,area,pie,scatter,composed}
            spec.chart_type ← "bar"

    8.  EXECUTE spec.sql against sql_ai.db
        rows ← execute(spec.sql)

    9.  COERCE numeric values
        FOR each row in rows:
            TRY cast row[y_key] to float

   10.  SAVE chart to app.db
        INSERT SavedChart(user_id, title, chart_type, color,
                          sql_query, x_key, y_key)

   11.  RETURN {
            status: "success",
            chart_type, title, description,
            x_key, y_key, color,
            x_label, y_label,
            executed_sql, data: rows,
            row_count, warning (if no data)
        }
  END

modified, and dropped by the user through natural-language prompts.

statements at runtime.  The schema is reflected dynamically via
SQLAlchemy's MetaData.reflect() before every AI interaction.

       correctly parsed.

                 7.  CONCLUSIONS AND FUTURE SCOPE
================================================================================

CONCLUSIONS
────────────

Invixa AI successfully demonstrates that modern Large Language Models can
be harnessed to create an intelligent, conversational interface for
database management and analytics — all running locally with full data
privacy.

The key achievements of this project are:

  1.  Natural-Language Database Operations:  Users can create, query,
      modify, and delete database structures and data using plain English,
      completely eliminating the need for SQL expertise.

  2.  AI-Powered Analytics:  The system generates accurate, interactive
      charts from natural-language descriptions, supporting six chart
      types (bar, line, area, pie, scatter, composed) with automatic
      chart type selection, axis mapping, and data coercion.

  3.  Secure Authentication:  A complete user authentication system with
      bcrypt password hashing, profile management, and password change
      functionality ensures secure multi-user access.

  4.  Real-Time Feedback:  The split-panel UI provides instant feedback —
      the database preview auto-refreshes after mutations, chat messages
      appear with typewriter animations, and charts render dynamically
      on a dedicated canvas.

  5.  Premium UI/UX:  The dark-themed, glassmorphism design with
      micro-animations, gradient accents, and responsive layouts delivers
      a polished, production-quality user experience.

  6.  Privacy-First Architecture:  All processing (LLM inference, database
      operations, authentication) runs on the user's local machine.  No
      data is transmitted to external cloud services.

  7.  Persistent History:  Generated charts and table metadata are saved
      in a dedicated application database, allowing users to revisit and
      re-render previous analytical work with live data.


FUTURE SCOPE
─────────────

While the current implementation meets all core objectives, the following
enhancements are planned for future iterations:

  1.  JWT-Based Authentication
      Replace the current stateless localStorage approach with JSON Web
      Tokens (JWT) for more robust session management, token expiry, and
      refresh token rotation.

  2.  Multi-Database Support
      Extend beyond SQLite to support PostgreSQL, MySQL, and MongoDB,
      allowing users to connect to external databases.

  3.  Export & Sharing
      Add the ability to export charts as PNG/SVG images and PDF reports.
      Allow sharing of chart configurations via shareable links.

  4.  Copy SQL to Clipboard
      Add a one-click "Copy SQL" button to the SQL badge component,
      as suggested during beta testing.

  5.  Conversational Memory
      Implement multi-turn conversation memory so the AI can reference
      previous prompts and results within the same session.

  6.  Role-Based Access Control
      Implement proper RBAC so that admin users can manage other users,
      restrict access to certain tables, and audit query history.

  7.  Collaborative Workspaces
      Allow multiple users to share a database workspace, with real-time
      synchronisation and collaborative querying.

  8.  Custom Model Selection
      Allow users to switch between different Ollama models from the UI,
      comparing quality and speed trade-offs.

  9.  Advanced Chart Customisation
      Add UI controls for chart customisation: custom colours, axis
      labels, gridlines, annotations, and multi-series overlays.

  10. Mobile-Optimised Interface
      Adapt the current desktop-focused layout for tablet and mobile
      devices with a responsive bottom-navigation pattern.

  11. Query History & Favourites
      Allow users to bookmark frequently used queries and access a
      searchable log of all past SQL operations.

  12. Data Import/Export
      Support CSV/JSON file upload to populate tables, and CSV/JSON
      export of query results.
